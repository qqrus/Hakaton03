# Карта взаимодействий (Agent World)

## Сущности

### Клиент (Web)
- Показывает карту мира, ленту событий, профиль агента и отношения.
- Получает обновления состояния и событий по WebSocket без перезагрузки страницы.
- Не генерирует события напрямую (лента меняется только из-за действий агентов).

### Сервер (API + Realtime)
- Отдаёт снимок текущего состояния мира по HTTP.
- Стримит события и состояние по WebSocket.
- Держит симуляционный движок, который запускается автоматически при старте.

### Симуляционный движок
- Поддерживает мир (локации, ресурсы, контроль территорий).
- Поддерживает агентов (цели, черты, ограничения, отношения, память).
- Каждый тик выбирает действие для каждого агента, применяет его к миру и публикует событие.
- Все изменения ленты событий происходят только через события, порождённые действием агента.

### LLM-слой (OpenRouter)
- Используется для выбора действий агентов и генерации текста сообщений.
- Ключи хранятся только на сервере (переменные окружения), ротируются по агентам.

### Память и отношения
- Память хранит эпизоды и краткие сводки, доступна для поиска по запросу.
- Отношения (affinity/trust) изменяются от сообщений, конфликтов и ключевых действий.

## Потоки данных

### Запуск приложения
1. Сервер стартует и поднимает WebSocket.
2. Движок стартует автоматически и сразу начинает тики.
3. Клиент подключается к WebSocket и получает первичный `state`.

### Тик симуляции (основной цикл)
1. Для каждого агента собирается контекст:
   - позиция, локальные ресурсы, контроль территории
   - ближайшие агенты, их фракции и отношения
   - последние события и релевантные воспоминания (memory search)
2. Агент выбирает действие через LLM (или fallback-стратегию).
3. Действие применяется к миру:
   - перемещение меняет позицию
   - добыча уменьшает ресурс в локации и усиливает контроль
   - переговоры/сообщения меняют отношения и создают события в ленте
   - конфликт меняет отношения и может менять контроль территории
4. Результат действия порождает `event`, который:
   - попадает в ленту
   - записывается в память участников
   - отражается в отношениях
5. Сервер транслирует `event` и обновлённый `state` в WebSocket.

### Обновление UI
- WebSocket сообщения:
  - `state`: полный снимок (на подключение и периодически)
  - `event`: инкрементальное событие для ленты (с последующим обновлением state)
- UI перерисовывает:
  - карту (контроль территорий и позиции агентов)
  - ленту (новые события сверху)
  - инспектор (изменения целей/настроения/отношений)

## Контракты сообщений

### WebSocket → Client
```json
{ "type": "state", "data": { "now": 0, "speed": 1, "world": {}, "agents": [], "relations": [], "feed": [] } }
```
```json
{ "type": "event", "data": { "id": "evt", "ts": 0, "type": "message", "title": "...", "text": "...", "importance": 0.5 } }
```

## Ограничения безопасности
- Ключи OpenRouter не должны попадать в репозиторий и не должны отдаваться клиенту.
- Любые административные операции должны требовать токен (Bearer).
- Логи не должны содержать секреты; в отладочном режиме допускается расширенная трассировка, но без ключей.

## Инварианты системы
- Любое событие в ленте должно быть привязано к конкретному действию агента (через участников/контекст).
- Клиент не имеет API для прямого добавления произвольных событий.
- Движок запускается автоматически и агенты активны сразу после старта.
